## 測試目標
確保使用者註冊與密碼重設功能符合規格需求，包含前後端驗證、API 正確行為、錯誤處理與安全性要求。

## 測試範圍
### In Scope
- 使用者註冊流程（前端格式驗證、API 請求與回應）
- 忘記密碼請求流程（Email Token 發送與回應）
- 密碼重設流程（Token 驗證與密碼更新）
- API 錯誤碼與限流機制驗證

### Out of Scope
- 第三方登入（Google/GitHub OAuth）
- 註冊階段 Email 驗證
- 舊版瀏覽器相容性

## 測試環境
- 測試伺服器環境，包含完整後端服務與資料庫
- 模擬 Email 發送服務
- 支援 HTTPS 的測試環境
- 可控限流與錯誤模擬機制

## 測試策略
- Unit Test：驗證前端輸入格式檢查函式、後端服務邏輯
- Integration Test：API 請求與資料庫交互
- E2E Test：模擬使用者完整註冊與密碼重設流程
- Regression Test：確保歷史功能不被破壞

## 測試流程設計
### 主流程
1. 使用者輸入註冊資料，前端格式驗證通過
2. 發送註冊 API 請求，後端驗證帳號唯一性
3. 成功建立帳號，回應 201，前端導向登入頁
4. 使用者忘記密碼，輸入 Email，發送忘記密碼請求
5. 後端產生 Token 並發送 Email，回應 200
6. 使用者點擊重設密碼連結，輸入新密碼
7. 後端驗證 Token，有效則更新密碼並回應成功

### 例外流程
- 註冊時帳號已存在，API 回傳 409，前端提示錯誤
- 註冊時輸入格式錯誤，API 回傳 400，前端顯示錯誤訊息
- 忘記密碼請求過於頻繁，API 回傳 429，前端提示限流訊息
- 密碼重設 Token 過期或無效，API 回傳 401/403，導向錯誤頁
- Email 寄送失敗，API 回傳 500，前端跳出錯誤視窗

## API 驗證重點
- Request Schema：
  - 註冊：nickname, username, password, confirm_password
  - 忘記密碼：email
  - 密碼重設：token, new_password, confirm_password
- Response Schema：
  - 成功訊息與錯誤訊息格式一致
- 主要錯誤碼：400, 401, 403, 409, 429, 500
